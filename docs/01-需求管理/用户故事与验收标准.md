# 用户故事与验收标准

**最后更新：** 2026-02-06

> 本文件用于补充“实现级验收口径”（例如 API/页面/数据检查点）。  
> 用户视角的验收要点以 `需求分析.md` 附录 B 为基线，本文件在不冲突的前提下做细化。

## 1. 全局验收规则

- 接口响应统一：`{ code, message, data, request_id }`。
- 每个 P0 接口至少包含成功/失败各 1 个自动化测试。
- 需求状态更新必须同步：`需求池.md`、`../05-迭代开发记录/`。
- 后续迭代需求可先采用“草案验收口径”，进入开发前需细化为可测试标准。
- 暂缓/候选需求不纳入当前迭代验收范围。

## 2. V1 验收定义（REQ-001~REQ-003，REQ-006）

### REQ-001 股票基础信息

用户故事：作为用户，我希望按代码或名称查询股票基础信息，并支持后台同步。

验收标准：
- `GET /stocks` 支持 `q/industry/page/page_size/sort`。
- `POST /stocks/sync` 支持请求体或默认股票列表同步。
- 返回字段包含 `code/name/industry/market`。

### REQ-002 自选分组

用户故事：作为用户，我希望创建、修改、删除分组，并维护分组内股票。

验收标准：
- 支持 `GET/POST/PUT/DELETE /watchlists`。
- 支持 `POST/DELETE /watchlists/{id}/stocks`。
- 重名分组、重复加股返回可识别错误。

### REQ-003 历史报告

用户故事：作为用户，我希望查看个股历史报告、全局历史报告和单份报告正文。

验收标准：
- 支持 `GET /stock/{code}/reports`。
- 支持 `GET /reports`（支持筛选条件）。
- 支持 `GET /reports/{id}` 并返回 markdown 正文。

### REQ-006 个股分析触发与报告生成

用户故事：作为用户，我希望手动触发分析并快速查看新生成报告。

验收标准：
- 提供 `POST /stock/{code}/analyze`。
- 返回 `report_id/markdown_path/generated_at`。
- 可通过历史报告接口查询到新报告记录。

## 3. 后续迭代验收草案（节选）

### REQ-004 基础访问控制（单用户）

用户故事：作为单用户，我希望系统具备基础访问控制，避免服务暴露导致他人访问我的数据。

草案验收口径：
- Web UI 和 API 默认需要访问凭证（例如访问密钥、简单登录等，方案二选一即可）。
- 未携带/校验失败返回 `401` 或给出统一的拒绝提示（具体口径统一即可）。
- 凭证支持配置与轮换（MVP 允许手动修改）。

### REQ-008 AI 对话与知识检索

用户故事：作为研究用户，我希望基于历史报告进行问答检索。

草案验收口径：
- 支持基于股票/时间范围检索报告片段。
- 对话结果可回溯到引用报告来源。

### REQ-009 持仓管理

用户故事：作为用户，我希望记录持仓并关联分析结果。

草案验收口径：
- 支持持仓新增、修改、删除。
- 支持按持仓展示关联报告摘要。

### REQ-010 盯盘告警

用户故事：作为用户，我希望配置阈值并在触发后收到告警。

草案验收口径：
- 支持价格或涨跌幅阈值配置。
- 支持通知渠道发送并记录告警日志。

### REQ-011 策略管理与策略执行

用户故事：作为进阶用户，我希望定义策略并查看策略执行结果。

草案验收口径：
- 支持策略定义与启停。
- 支持策略结果记录与可追溯查询。
