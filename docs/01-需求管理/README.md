# 需求管理

## 目标

将产品需求统一收敛到可执行、可追踪、可验收的文档集合，先形成整体需求全景，再按 MVP、Iteration、Frozen 分层推进。

## 单一事实源

- 主基线：`2026-02-06-从0到1-需求分析.md`
- 若与其他需求文档冲突，以主基线为准，并在 `需求变更记录.md` 记录变更。

## 文档清单

- `2026-02-06-从0到1-需求分析.md`：当前生效需求基线（含整体需求全景 + 分层范围）。
- `需求池.md`：需求排期与当前状态看板（含冻结需求）。
- `用户故事与验收标准.md`：按 REQ 编号定义验收口径（MVP 详细、后续草案、冻结草案）。
- `需求变更记录.md`：需求变更的原因、影响、回滚记录。
- `需求评审清单.md`：需求评审前后检查项。

## 使用流程

1. 先在主基线中确认需求归属：`MVP`、`Iteration` 或 `Frozen`。
2. 新需求进入 `需求池.md`，分配 ID（`REQ-xxx`/`NFR-xxx`）。
3. 评审通过后补充用户故事与验收标准。
4. 实施前必须绑定：
   - 设计文档：`../02-系统设计/`
   - 实施计划：`../04-实施计划/`
5. 需求变更必须登记到 `需求变更记录.md`。
6. 迭代结束回写状态并同步 `../05-迭代开发记录/`。

## 状态定义

- `backlog`：已识别但未排期。
- `todo`：已提出，待评审或待开发。
- `planned`：已确认，待进入开发。
- `in-progress`：开发中。
- `blocked`：受阻。
- `on-hold`：冻结，暂不考虑。
- `done`：实现与验收完成。
