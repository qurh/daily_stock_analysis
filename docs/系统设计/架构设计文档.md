## 3. 产品哲学与系统原则

### 3.1 投资认知状态（Cognition Layer）

系统需要有一个**显式的、可被审视的认知立场**，而非将判断逻辑隐含在 Prompt 中。

#### 3.1.1 认知状态模型

| 维度 | 说明 | 示例 |
|------|------|------|
| 市场阶段判断 | 当前市场所处的阶段 | 震荡、趋势、极端、底部、顶部 |
| 系统风险态度 | 当前系统的风险偏好设置 | 保守、中性、激进 |
| 偏好权重 | 技术面、基本面、消息面的权重分配 | 技术 0.5、基本面 0.3、消息 0.2 |
| 置信度等级 | 当前判断的确信程度 | 高、中、低 |

#### 3.1.2 认知状态的作用

所有分析、建议、告警**必须引用当前的认知状态**：

* 当技术面强、基本面弱、情绪极端时 → 系统基于风险态度给出「谨慎博弈」建议
* 当长期趋势向上、短期破位时 → 系统基于投资风格给出「分批建仓」或「止损离场」建议

### 3.2 系统核心原则

| 原则 | 说明 |
|------|------|
| 诚实原则 | 系统可以输出「当前无法给出明确结论」 |
| 可追溯原则 | 所有结论必须可追溯到具体的数据和推理 |
| 可修正原则 | 结论失效时自动标记，而非悄悄过期 |
| 用户主权原则 | 用户可以配置系统的建议级别和风险偏好 |

### 3.3 投资风格配置

用户可选择或自定义系统风格：

| 风格 | 描述 | 适用场景 |
|------|------|----------|
| 趋势跟踪 | 顺势而为，趋势不破不走 | 牛市、趋势行情 |
| 价值投资 | 关注基本面，等待估值回归 | 长期投资、左侧交易 |
| 短线博弈 | 快进快出，捕捉短期波动 | 震荡市、题材炒作 |
| 保守稳健 | 严控风险，优先保本 | 新手、风险厌恶型 |

### 3.4 责任边界协商

系统需要明确与用户之间的**责任边界**，让用户心理有底。

#### 3.4.1 建议级别

| 级别 | 说明 | 用户控制 |
|------|------|----------|
| 信息型 | 仅提供客观数据和分析 | 默认开启 |
| 分析型 | 提供多角度分析和风险提示 | 默认开启 |
| 行动建议型 | 给出明确的买卖建议 | 可关闭 |
| 自动执行型 | 自动执行交易（需要额外授权） | 需手动开启 |

#### 3.4.2 用户可配置项

| 配置项 | 说明 |
|--------|------|
| 是否允许系统给出明确买卖建议 | 关闭后只输出风险提示 |
| 是否接收盯盘告警推送 | 可单独关闭某类告警 |
| 置信度阈值 | 低于阈值的结论不主动推送 |
| 风险偏好 | 影响系统建议的激进程度 |

## 4. 认知与决策层

本节定义系统如何处理**不确定性、冲突信息与复杂判断**。

### 4.1 冲突信息处理策略

真实投资世界充满矛盾信号，系统需要明确的冲突处理机制。

#### 4.1.1 冲突类型

| 类型 | 示例 |
|------|------|
| 指标冲突 | MA5 金叉但 MACD 死叉 |
| 周期冲突 | 长期趋势向上但短期破位 |
| 数据源冲突 | A 数据源显示利好，B 数据源显示利空 |
| 逻辑冲突 | 技术面看多但基本面看空 |

#### 4.1.2 冲突解决方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| 加权投票 | 根据权重计算综合得分 | 多指标综合判断 |
| 延迟判断 | 等待更多数据确认 | 趋势临界点 |
| 多结论输出 | 输出「看涨情景」和「看跌情景」 | 关键决策点 |
| 明确声明无法判断 | 承认不确定性 | 信息严重不足 |

### 4.2 置信度机制

系统输出的所有结论都应附带**置信度标签**。

#### 4.2.1 置信度等级

| 等级 | 分值范围 | 说明 |
|------|----------|------|
| 高 | 0.8 - 1.0 | 多维度验证，判断确定性高 |
| 中 | 0.5 - 0.8 | 有一定依据，但存在不确定性 |
| 低 | 0.2 - 0.5 | 依据不足，仅供参考 |
| 无法判断 | < 0.2 | 信息不足或冲突严重 |

#### 4.2.2 影响置信度的因素

| 因素 | 影响 |
|------|------|
| 数据新鲜度 | 越新的数据置信度越高 |
| 数据源质量 | 权威数据源置信度高 |
| 指标一致性 | 多指标一致置信度高 |
| 历史准确率 | 历史表现好的模型置信度高 |

### 4.3 时间感知（Temporal Awareness）

投资决策高度依赖时间维度，系统需要明确结论的**时间有效性**。

#### 4.3.1 结论有效期分类

| 类型 | 有效期 | 说明 |
|------|--------|------|
| 即时判断 | 盘中有效 | 价格波动即失效 |
| 日内判断 | 当日有效 | 收盘后需重新评估 |
| 周度判断 | 本周有效 | 周末后需重新评估 |
| 长期判断 | 月度以上 | 趋势未变则持续有效 |

#### 4.3.2 事实、判断、推测的区分

| 类型 | 特征 | 示例 |
|------|------|------|
| 事实 | 不会变的数据 | 「MA5=1750」 |
| 判断 | 会随市场变化的分析 | 「短期趋势向上」 |
| 推测 | 不确定的预测 | 「明日可能上涨」 |

系统输出时需**明确标注类型**，帮助用户理解结论的可变性。

## 5. 演进与反思层

本节定义系统如何**从错误中学习、持续自我优化**。

### 5.1 反例库（Failure Cases Library）

系统不仅要积累成功经验，更要系统性地记录**失败案例**。

#### 5.1.1 反例库定位

反例库是知识库的重要组成部分，与成功案例**同等重要**：

* 记录系统判断错误的案例
* 分析错误的原因（数据延迟、指标失效、市场结构变化等）
* 提取可复用的教训
* 在类似场景下自动引用历史教训

#### 5.1.2 反例记录内容

| 字段 | 说明 |
|------|------|
| 案例 ID | 唯一标识 |
| 原始结论 | 系统当时给出的判断 |
| 实际情况 | 后续市场的真实走势 |
| 错误类型 | 数据延迟、指标失效、逻辑漏洞等 |
| 教训总结 | 从该案例中学到的经验 |
| 改进措施 | 对系统的调整建议 |

#### 5.1.3 反例库使用

当系统给出类似判断时，可以引用：

```
「注意：2024 年 Q4 出现过类似技术形态，
随后出现大幅回调。当时的教训是：xxx
当前置信度已下调至中等偏保守。」
```

### 5.2 结论生命周期管理

#### 5.2.1 结论状态

| 状态 | 说明 |
|------|------|
| 初步判断 | 刚生成的结论 |
| 条件成立 | 触发条件已验证 |
| 已失效 | 条件被否定或市场已变化 |
| 已验证 | 经过时间验证的正确结论 |

#### 5.2.2 结论失效触发条件

| 条件类型 | 示例 |
|----------|------|
| 价格触发 | 止损位被击穿 |
| 时间触发 | 日内结论收盘后失效 |
| 事件触发 | 突发公告改变基本面 |
| 指标触发 | 关键指标出现反向信号 |

#### 5.2.3 自动标记机制

系统定期检查活跃结论，对失效结论自动标记为「已失效」，避免用户参考过期信息。

### 5.3 系统自我修正机制

#### 5.3.1 修正触发条件

| 条件 | 说明 |
|------|------|
| 连续错误 | 连续 N 次判断错误 |
| 重大偏差 | 单次判断与实际偏差超过阈值 |
| 用户反馈 | 用户标记结果为错误 |

#### 5.3.2 修正行动

| 行动 | 说明 |
|------|------|
| 降低置信度 | 自动下调类似场景的置信度 |
| 调整权重 | 调整指标的权重配置 |
| 标记待审 | 标记需要人工审查的判断逻辑 |
| 记录反例 | 存入反例库供后续分析 |

### 5.4 知识资产的自我进化

#### 5.4.1 进化循环

```
每日复盘 → 提炼教训 → 更新知识库 → 影响未来判断 → 验证效果 → 再次复盘
```

#### 5.4.2 进化追踪

| 追踪项 | 说明 |
|--------|------|
| 知识使用频率 | 哪些知识被引用最多 |
| 知识准确率 | 被引用知识的验证结果 |
| 策略有效性 | 策略信号的成功率变化 |
| 系统整体表现 | 长期胜率、回撤控制等 |

## 7. 数据库设计

### 7.1 数据库选型

| 数据库 | 用途 | 扩展 |
|--------|------|------|
| PostgreSQL | 业务数据存储 | 主流关系型数据库 |
| pgvector | 向量索引 | RAG 语义搜索 |
| Redis（可选） | 缓存、任务队列 | 会话、实时数据 |
| 文件系统 | 文档存储 | Markdown、报告 |

## 10. 安全设计

### 10.1 敏感数据加密

```python
# 使用 Fernet 对称加密
from cryptography.fernet import Fernet

class Encryptor:
    def __init__(self):
        self.key = os.getenv("ENCRYPTION_KEY").encode()
        self.fernet = Fernet(self.key)

    def encrypt(self, data: str) -> str:
        return self.fernet.encrypt(data.encode()).decode()

    def decrypt(self, encrypted: str) -> str:
        return self.fernet.decrypt(encrypted.encode()).decode()
```

### 10.2 API Key 存储

- API Key 加密后存储在 `cfg_ai_models.api_key_encrypted`
- 运行时动态解密
- 日志中不输出 Key
