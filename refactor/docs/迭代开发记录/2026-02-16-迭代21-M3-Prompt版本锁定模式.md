## 1. 本迭代目标

1. 为策略 `prompt_refs` 增加版本锁定模式（strict/lenient）。
2. 支持“全局默认 + 绑定级覆盖”的锁定策略。
3. 在聊天链路提供可观测且可控的回退行为。

## 2. 计划范围（Plan）

1. 新增严格模式失败场景测试与覆盖模式测试。
2. 增加按指定版本渲染 Prompt 的服务能力。
3. 在 ChatService 落地 strict/lenient 路由规则并打通配置。

## 3. 实际完成（Done）

1. 配置能力：
   - 新增环境变量 `PROMPT_REF_LOCK_MODE`，默认 `lenient`
2. 绑定能力：
   - `POST /api/v2/strategy/{strategy_id}/bind` 支持 `prompt_lock_mode` 字段
   - 绑定详情/查询返回 `prompt_lock_mode`
3. Prompt 渲染能力：
   - `PromptService.render_prompt_version(prompt_id, version, variables)`
4. Chat 路由规则：
   - `strict`：绑定 `prompt_refs` 必须成功解析，否则返回冲突
   - `lenient`：先尝试绑定版本，失败后回退到 active/default/builtin
5. 错误映射：
   - prompt 锁定冲突映射为 HTTP `409`
6. 新增测试：
   - strict 模式下绑定版本不可用时请求失败
   - 绑定级 `lenient` 覆盖全局 `strict`

## 4. 未完成项（Not Done）

1. 暂未提供 `strict` 失败时的细粒度错误码枚举与失败候选列表。
2. 暂未扩展到分析链路的 Prompt 版本锁定策略。

## 7. 风险与问题

1. 风险描述：`prompt_refs` 使用非法版本格式时当前采用跳过策略，诊断信息有限。
2. 影响范围：线上排障效率。
3. 缓解措施：下阶段补结构化错误详情与审计日志。

## 8. 关键决策

1. 决策内容：默认 `lenient` 保持兼容，`strict` 通过配置/绑定显式启用。
2. 决策原因：先兼顾稳定性，再提供强一致控制能力。
3. 影响模块：Strategy Binding、Prompt Center、Chat、Config。
