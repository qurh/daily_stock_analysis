## 1. 本迭代目标

1. 在真实 API 联调中打通 `publish -> bind -> rollback` 正向链路。
2. 验证此前 `publish=409` 是否仅为业务门禁命中，而非接口缺陷。
3. 输出可复现的“可发布”数据构造路径。

## 2. 根因确认

经复现与代码追踪，`publish=409` 的触发条件来自策略发布门禁：

1. 发布必须有 `backtest_job_id`。
2. 回测门禁要求：
   - `sample_size >= 5`
   - `win_rate_pct >= 50`
   - 回测状态为 `completed` 或 `partial_completed`

此前失败信息：
- `STR-GATE-005: strategy did not pass backtest gate`

结论：该 409 为业务规则生效，非接口错误。

## 3. 联调执行过程（正向）

测试环境：
1. `LLM_PROVIDER=mock-llm`
2. 临时 `DATABASE_URL` + 临时 `CHROMA_PATH`

执行步骤：
1. 创建 chat session，发送消息；
2. `distill -> review(approve)` 生成并索引认知；
3. `extract strategy` 生成候选策略；
4. 按 symbol 批量构造分析样本（每个 symbol 生成 5 条 analysis）；
5. 对每个 symbol 提交 `backtest scope=symbol`，直到命中 `win_rate >= 50`；
6. 用命中的 `backtest_job_id` 调用发布；
7. 发布成功后执行 `bind`，再执行 `rollback`。

## 4. 结果

命中可发布回测样本：
1. `symbol=PUBLISH005`
2. `sample_size=5`
3. `win_rate_pct=60.0`
4. `backtest_job_id=61c44470-5223-44c9-bb94-29f39f8d5bf5`

链路结果：
1. `POST /strategy/{strategy_id}/publish` -> `200`
2. `POST /strategy/{strategy_id}/bind` -> `201`
3. `GET /strategy/bindings?flow_id=chat_reply_v1` -> `200`
4. `POST /strategy/{strategy_id}/rollback` -> `200`

结论：正向发布链路已联调通过。

## 5. 风险与建议

1. 当前“可发布样本”在 mock 场景下存在概率波动（与分析样本组合相关）。
2. 建议后续补一个“联调专用确定性样本构造器”（测试 helper 或 seed 脚本），降低联调随机性。
3. 前端 Strategy 页面可增加“发布前门禁状态预检”提示，减少 409 试错。
