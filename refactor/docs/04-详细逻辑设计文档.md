## 1. 文档目标

本文档用于将架构设计落到可编码、可测试、可验收的模块逻辑层，覆盖：

1. 按模块定义对外与对内接口。
2. 明确关键对象状态机和状态迁移约束。
3. 明确核心数据结构（请求、响应、实体、事件）。
4. 明确异常与错误码设计。
5. 给出模块级测试点，支持后续 TDD 和回归。

---

## 2. 设计约定

## 2.1 模块边界

本次详细设计覆盖 12 个模块：

1. `Interaction Gateway`
2. `Workflow Engine (LangGraph Adapter)`
3. `Prompt Center`
4. `Knowledge Service`
5. `Memory Service`
6. `Agent Toolkit`
7. `Analysis Service`
8. `Strategy Service`
9. `Backtest Service`
10. `Feedback & Optimization Service`
11. `Notification Hub`
12. `Governance & Observability`

## 2.2 接口分层约定

1. 外部接口：REST/SSE/WebSocket，前缀统一 `/api/v2`。
2. 应用服务接口：模块间同步调用（Python service interface）。
3. 领域事件接口：模块间异步解耦（Event Bus）。

## 2.3 状态机约定

1. 状态使用大写英文常量。
2. 状态迁移必须是显式触发，不允许隐式跨级跳转。
3. 任一状态机必须定义终态与失败终态。
4. 所有关键迁移必须写审计日志。

## 2.4 错误码约定

格式：`<MODULE>-<SCENE>-<NNN>`

示例：

- `WFE-VALIDATE-001`：工作流模板校验失败
- `KB-INGEST-004`：向量化阶段失败
- `FDB-GATE-002`：优化变更未通过评估闸门

## 2.5 测试分层约定

1. 单元测试：纯函数、状态机、转换器、校验器。
2. 集成测试：数据库/向量库/消息总线/外部依赖适配。
3. 端到端测试：从 API 到报告输出与通知投递。
4. 评估测试：Prompt/Flow/Strategy 变更前的离线评测。

## 4. 模块详细设计

## 4.1 Interaction Gateway

需求映射：`FR-CHAT-001~005`、`FR-KB-004`、`FR-BT-004`、`NFR-OBS-001`

### 4.1.1 对外接口

1. `POST /api/v2/chat/sessions`
2. `POST /api/v2/chat/sessions/{session_id}/messages`
3. `GET /api/v2/chat/sessions/{session_id}/messages`
4. `POST /api/v2/analysis/jobs`
5. `GET /api/v2/jobs/{job_id}`
6. `GET /api/v2/jobs/{job_id}/stream`（SSE）
7. `POST /api/v2/optimization/proposals`（支持 chatbot 间接触发）

### 4.1.2 对内接口

1. `SessionAppService.create_session(user_ctx) -> session_id`
2. `ChatAppService.handle_message(session_id, message, request_ctx) -> stream_handle`
3. `JobAppService.submit(job_type, payload, request_ctx) -> job_id`

### 4.1.3 状态机

`ChatSession`：

- `CREATED -> ACTIVE -> IDLE -> CLOSED`
- 异常迁移：`ACTIVE -> ARCHIVED`（超期或隐私删除触发）

`ApiJob`：

- `PENDING -> RUNNING -> SUCCEEDED`
- `PENDING|RUNNING -> FAILED|CANCELLED|TIMEOUT`

### 4.1.4 数据结构

- `ChatSessionDTO(session_id, user_id, memory_policy, status, created_at, updated_at)`
- `ChatMessageDTO(message_id, session_id, role, content, citations, tool_trace, created_at)`
- `ApiJobDTO(job_id, job_type, status, progress, result_ref, error_code, created_at)`

### 4.1.5 异常设计

- `IGW-SESSION-001`：session 不存在
- `IGW-REQUEST-002`：请求参数不合法
- `IGW-LIMIT-003`：触发频控
- `IGW-STREAM-004`：流式连接中断

### 4.1.6 测试点

1. 创建会话后多轮上下文是否正确继承。
2. SSE 是否按状态推送 `PENDING/RUNNING/SUCCEEDED/FAILED`。
3. 非法 session、重复提交、限流场景错误码是否准确。
4. chatbot 触发优化提案是否落到统一接口。

## 4.2 Workflow Engine（LangGraph Adapter）

需求映射：`FR-ORC-001~006`、`FR-AGT-001~004`、`NFR-MTN-004`

### 4.2.1 对外接口

1. `POST /api/v2/workflows/templates`
2. `POST /api/v2/workflows/templates/{flow_id}/publish`
3. `POST /api/v2/workflows/executions`
4. `GET /api/v2/workflows/executions/{execution_id}`
5. `POST /api/v2/workflows/executions/{execution_id}/cancel`

### 4.2.2 对内接口

1. `WorkflowEngineAdapter.validate(template) -> ValidationReport`
2. `WorkflowEngineAdapter.compile(template, version) -> ExecutableGraph`
3. `WorkflowEngineAdapter.run(executable_graph, input_ctx) -> execution_id`
4. `WorkflowEngineAdapter.resume(execution_id) -> status`
5. `WorkflowEngineAdapter.cancel(execution_id) -> status`

### 4.2.3 状态机

`FlowTemplateVersion`：

- `DRAFT -> VALIDATED -> PUBLISHED -> DEPRECATED`
- 异常迁移：`VALIDATED -> DRAFT`（变更后回退重验）

`FlowExecution`：

- `PENDING -> RUNNING -> SUCCEEDED`
- `RUNNING -> PARTIAL_FAILED -> SUCCEEDED|FAILED`
- `PENDING|RUNNING|PARTIAL_FAILED -> CANCELLED|TIMEOUT|FAILED`

### 4.2.4 数据结构

```json
{
  "flow_id": "stock_analysis_v1",
  "version": "1.2.0",
  "entry_node": "node_fetch_market",
  "nodes": [
    {
      "id": "node_fetch_market",
      "type": "data.fetch.market",
      "timeout_sec": 10,
      "retry": {"max_attempts": 2, "backoff_sec": 2},
      "next": ["node_fetch_macro", "node_fetch_credit"]
    }
  ],
  "guards": [
    {"if": "risk_level >= HIGH", "then": "node_risk_alert"}
  ]
}
```

- `NodeExecutionRecord(node_id, status, start_at, end_at, input_ref, output_ref, error_code)`
- `ExecutionTrace(execution_id, flow_version, prompt_versions, tool_calls, metrics)`

### 4.2.5 异常设计

- `WFE-VALIDATE-001`：模板缺失入口节点
- `WFE-VALIDATE-002`：图结构存在环路/死路
- `WFE-RUN-003`：节点契约不匹配
- `WFE-RUN-004`：执行超时
- `WFE-RUN-005`：恢复执行失败

### 4.2.6 测试点

1. 串行、并行、条件分支是否按预期执行。
2. 节点失败后重试/降级路径是否生效。
3. 执行 trace 是否包含 flow/prompt/tool 版本信息。
4. LangGraph 适配层替换 mock engine 时契约是否保持。

- `DRAFT -> TESTED -> EVALUATED -> APPROVED -> CANARY -> ACTIVE`

### 4.3.5 异常设计

- `PRM-RENDER-001`：变量缺失
- `PRM-SCHEMA-002`：输出 schema 不匹配
- `PRM-EVAL-003`：评估任务失败
- `PRM-GATE-004`：未通过评估不可发布

### 4.3.6 测试点

1. 变量渲染和模板拼接正确性。
2. 版本回滚后是否即时生效。
3. 不同环境（dev/staging/prod）是否隔离。
4. 评估失败版本是否被拦截发布。

## 4.4 Knowledge Service

需求映射：`FR-KB-001~006`、`NFR-PERF-002`  
协作需求：`FR-KB-008`（由 Strategy Service 主责，Knowledge Service 负责审核通过后的入库与索引执行）

### 4.4.1 对外接口

1. `POST /api/v2/knowledge/documents/upload`
2. `POST /api/v2/knowledge/documents/{doc_id}/optimize`
3. `POST /api/v2/knowledge/documents/{doc_id}/ingest`
4. `GET /api/v2/knowledge/documents/{doc_id}`
5. `GET /api/v2/knowledge/chunks/search`
6. `DELETE /api/v2/knowledge/documents/{doc_id}`

### 4.4.2 对内接口

1. `IngestionPipeline.run(doc_id, options) -> job_id`
2. `ChunkService.chunk(markdown, policy) -> List[Chunk]`
3. `EmbeddingService.embed(chunks, model) -> vectors`
4. `RetrievalService.search(query, top_k, filters) -> HitChunks`

### 4.4.3 状态机

`IngestionJob`：

- `CREATED -> UPLOADED -> CLEANED -> CHUNKED -> SUMMARIZED -> EMBEDDED -> INDEXED -> COMPLETED`
- 任意中间态可迁移到 `FAILED`
- 可重试迁移：`FAILED -> RETRYING -> <failed_state_next>`

### 4.4.4 数据结构

- `KnowledgeDocument(doc_id, title, source_type, raw_uri, optimized_uri, status, tags)`
- `KnowledgeChunk(chunk_id, doc_id, section_path, content, summary, token_count, embedding_ref)`
- `IngestionJob(job_id, doc_id, stage, progress, error_code, retry_count)`

### 4.4.5 异常设计

- `KB-UPLOAD-001`：文件格式不支持
- `KB-CLEAN-002`：Markdown 结构清洗失败
- `KB-CHUNK-003`：分块策略冲突
- `KB-EMBED-004`：向量化失败
- `KB-INDEX-005`：索引构建失败

### 4.4.6 测试点

1. 上传到入库全链路成功路径。
2. 大文档分块边界与来源回溯准确性。
3. 单文档失败不阻塞批任务。
4. 检索返回是否带来源证据和定位信息。

## 4.10 Feedback & Optimization Service

需求映射：`FR-FDB-001~005`、`FR-ORC-006`、`FR-STR-005`

### 4.10.1 对外接口

1. `POST /api/v2/feedback/records`
2. `GET /api/v2/feedback/records`
3. `POST /api/v2/optimization/jobs/trigger`
4. `POST /api/v2/optimization/proposals`
5. `POST /api/v2/optimization/proposals/{proposal_id}/approve`
6. `POST /api/v2/optimization/proposals/{proposal_id}/reject`

### 4.10.2 对内接口

1. `FeedbackService.record(feedback) -> feedback_id`
2. `OptimizationService.trigger(trigger_ctx) -> opt_job_id`
3. `OptimizationService.evaluate_change(proposal_id) -> EvalReport`
4. `OptimizationService.rollout(proposal_id, strategy) -> rollout_id`

### 4.10.3 状态机

`ChangeProposal`：

- `CREATED -> TRIAGED -> SIMULATED -> REVIEW_PENDING -> APPROVED -> ROLLING_OUT -> DEPLOYED`
- `SIMULATED|REVIEW_PENDING -> REJECTED`
- `ROLLING_OUT -> ROLLED_BACK`

`OptimizationJob`：

- `QUEUED -> RUNNING -> EVALUATED -> COMPLETED`
- `RUNNING|EVALUATED -> FAILED`

### 4.10.4 数据结构

- `FeedbackRecord(feedback_id, target_id, score, tags, comment, source, created_at)`
- `OptimizationFeatureSet(feature_set_id, data_ref, sample_size, generated_at)`
- `ChangeProposal(proposal_id, source, target, diff, status, gate_result, reviewer)`
- `RolloutRecord(rollout_id, target_ref, strategy, ratio, status, metrics_snapshot)`

### 4.10.5 异常设计

- `FDB-INPUT-001`：反馈输入不合法
- `FDB-TRIGGER-002`：触发条件不满足
- `FDB-GATE-003`：评估闸门未通过
- `FDB-REVIEW-004`：审批状态冲突
- `FDB-ROLLOUT-005`：灰度发布失败

### 4.10.6 测试点

1. 事件驱动触发是否按规则命中。
2. 手动触发和 chatbot 触发是否走统一提案流程。
3. 未审批提案是否被禁止发布。
4. 灰度失败是否自动回滚。

## 4.11 Notification Hub

需求映射：`FR-NTF-001~003`、`NFR-REL-001`

### 4.11.1 对外接口

1. `POST /api/v2/notifications/preview`
2. `POST /api/v2/notifications/send`
3. `GET /api/v2/notifications/channels`
4. `POST /api/v2/notifications/channels/test`

### 4.11.2 对内接口

1. `NotificationHub.dispatch(message, policy, context) -> DeliveryReport`
2. `ChannelPlugin.send(payload, context) -> DeliveryResult`
3. `NotificationFormatter.render(channel, report, policy) -> rendered_payload`

### 4.11.3 状态机

`DeliveryTask`：

- `CREATED -> RENDERED -> DISPATCHING -> DELIVERED`
- `DISPATCHING -> PARTIAL_DELIVERED`
- `DISPATCHING -> FAILED -> RETRYING -> DISPATCHING`

### 4.11.4 数据结构

- `NotificationMessage(message_id, title, content, attachments, priority)`
- `ChannelPolicy(channel, enabled, format, max_bytes, retry_policy)`
- `DeliveryRecord(delivery_id, message_id, channel, status, attempt, error_code, sent_at)`

### 4.11.5 异常设计

- `NTF-CHANNEL-001`：渠道未启用
- `NTF-FORMAT-002`：渠道格式渲染失败
- `NTF-SEND-003`：发送失败
- `NTF-RETRY-004`：重试耗尽

### 4.11.6 测试点

1. 全渠道并发发送与结果聚合。
2. 长消息分片策略与 Markdown 方言适配。
3. 单渠道失败不影响其他渠道投递。
4. 新插件接入不修改核心分发代码。

## 4.12 Governance & Observability

需求映射：`NFR-OBS-001~004`、`NFR-SEC-001~003`、`NFR-MTN-003`

### 4.12.1 对外接口

1. `GET /api/v2/observability/traces/{trace_id}`
2. `GET /api/v2/observability/metrics`
3. `GET /api/v2/audit/logs`
4. `POST /api/v2/governance/policies/{policy_id}/apply`

### 4.12.2 对内接口

1. `TraceService.start_span(ctx) -> span_id`
2. `MetricService.emit(metric_name, tags, value) -> None`
3. `AuditService.record(action, actor, target, diff) -> audit_id`
4. `GovernanceService.enforce_gate(target, eval_report) -> GateDecision`

### 4.12.3 状态机

`PolicyRollout`：

- `PLANNED -> CANARY -> OBSERVING -> FULLY_ROLLED_OUT`
- `CANARY|OBSERVING -> HALTED -> ROLLED_BACK`

### 4.12.4 数据结构

- `TraceSpan(span_id, trace_id, parent_id, module, start_at, end_at, status)`
- `MetricPoint(metric, value, tags, ts)`
- `AuditLog(audit_id, actor, action, target, before, after, reason, created_at)`
- `GateDecision(target_ref, pass, thresholds, notes, reviewer, decided_at)`

### 4.12.5 异常设计

- `GOV-AUDIT-001`：审计写入失败
- `GOV-METRIC-002`：指标上报失败
- `GOV-GATE-003`：治理闸门判定异常
- `GOV-POLICY-004`：策略应用冲突

### 4.12.6 测试点

1. 单次请求 trace 是否贯通 API->Flow->Tool->LLM->Storage。
2. 关键变更是否落审计并可查询。
3. 治理闸门失败时是否阻断发布。
4. 指标采集异常是否不影响主链路。

## 4.13 模块主责归属说明

原则：每个需求 ID 仅定义一个主责模块，其他模块按协作职责实现。

| 需求 ID | 主责模块 | 协作模块 | 说明 |
|---|---|---|---|
| `FR-KB-008` | `Strategy Service` | `Knowledge Service` | Strategy 负责审核流与状态流转，Knowledge 负责审核通过后的入库与索引 |
| `FR-ORC-005` | `Strategy Service` | `Workflow Engine`, `Prompt Center` | Strategy 产出反哺策略，Workflow/Prompt 负责执行落地 |
| `FR-BT-003` | `Backtest Service` | `Feedback & Optimization Service` | Backtest 输出评估特征，Feedback 负责消费并触发优化 |

## 5. 关键跨模块状态机汇总

| 对象 | 状态机主链路 | 失败链路 |
|---|---|---|
| FlowExecution | `PENDING -> RUNNING -> SUCCEEDED` | `RUNNING -> FAILED/CANCELLED/TIMEOUT` |
| PromptVersion | `DRAFT -> TESTED -> EVALUATED -> APPROVED -> CANARY -> ACTIVE` | `CANARY/ACTIVE -> ROLLED_BACK` |
| StrategyVersion | `CANDIDATE -> EVALUATING -> APPROVED -> CANARY -> ACTIVE` | `EVALUATING -> REJECTED`, `CANARY/ACTIVE -> ROLLED_BACK` |
| BacktestJob | `CREATED -> QUEUED -> RUNNING -> SUMMARIZING -> COMPLETED` | `RUNNING -> FAILED` |
| ChangeProposal | `CREATED -> TRIAGED -> SIMULATED -> REVIEW_PENDING -> APPROVED -> ROLLING_OUT -> DEPLOYED` | `SIMULATED/REVIEW_PENDING -> REJECTED`, `ROLLING_OUT -> ROLLED_BACK` |
| IngestionJob | `UPLOADED -> CLEANED -> CHUNKED -> SUMMARIZED -> EMBEDDED -> INDEXED -> COMPLETED` | `<ANY_STAGE> -> FAILED -> RETRYING` |
