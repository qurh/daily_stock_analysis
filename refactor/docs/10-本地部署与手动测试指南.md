# 10-本地部署与手动测试指南

## 1. 目标与范围

本指南用于在本地完成重构项目（`refactor`）的：

1. 前后端部署启动
2. 基础可用性验证
3. 手动功能联调测试

仅针对重构目录：

- 后端：`refactor/backend`
- 前端：`refactor/frontend`

不要混用原项目根目录下的 `.env`。重构后端只读取 `refactor/backend/.env`（或脚本白名单加载的变量）。

---

## 2. 环境要求

建议版本：

1. Python `3.10+`
2. Node.js `18+`
3. npm `9+`
4. 操作系统：macOS / Linux

可选工具：

1. `curl`（接口验活）
2. `jq`（查看 JSON）

---

### 3.3 模式A：本地快速联调（mock LLM，推荐先跑通）

在 `refactor/backend/.env` 至少确认这些项：

启动：

```bash

source .venv/bin/activate

```

验活：

```bash
curl -s http://127.0.0.1:18000/api/v2/health
```

期望返回：

```json
{"status":"ok","timestamp":"..."}
```

### 3.4 模式B：真实模型联调（DashScope/千问）


其他建议保持：


启动命令同上（`--port 18000`）。

## 5. 手动测试清单（建议顺序）

### 5.1 基础连通性

1. 打开前端页面，确认路由可见（Chat/Knowledge/Workflow/Strategy/Backtest）。
2. 若页面报网络错误，先确认后端健康接口可用。

### 5.2 Knowledge 页面

1. 上传 Markdown 文档（title + markdown）。
2. 执行 optimize / ingest。
3. 执行 chunk 搜索。
4. 期望：可看到 `doc_id`、`chunk_count`、搜索结果命中。

### 5.3 Chat 页面

1. 创建 session。
2. 发送消息。
3. 查询消息列表。
4. 期望：assistant 返回内容，`tool_trace` 中可看到 `agent_trace`（如果命中工具）。

### 5.4 Workflow 页面

1. 启动 execution（`flow_id=stock_analysis_v1`）。
2. 查询 execution。
3. 执行 cancel。
4. 期望：能看到 trace nodes 和状态变更。

### 5.5 Backtest 页面

1. 创建 backtest job（`scope=market` 或 `scope=symbol`）。
2. 查询 job 状态。
3. 查看 results / performance。
4. 期望：有 `sample_size`、`win_rate_pct` 等指标。

### 5.6 Strategy 页面（核心）

1. `distill cognition`
2. `review memo`（approve）
3. `extract strategy`
4. `publish strategy`
5. `bind strategy`
6. `rollback strategy`

说明：

1. 若 publish 返回 `409 STR-GATE-005`，页面会显示门禁提示（样本数与胜率要求）。
2. 这是业务门禁，不是接口故障。

## 7. 常见问题排查

### 7.1 端口被占用

现象：`Address already in use`

处理：

```bash
lsof -i :18000
lsof -i :5173
kill -9 <PID>
```

---

### 7.2 前端报网络错误

优先检查：

1. `VITE_API_BASE_URL` 是否指向正确端口
2. 后端是否已启动并健康
3. 前端是否用了旧缓存（重启 `npm run dev`）

---

### 7.3 Strategy 发布 409

常见返回：`STR-GATE-005: strategy did not pass backtest gate`

原因：

1. `sample_size < 5` 或 `win_rate_pct < 50`

建议：

1. 多生成 analysis 样本后重跑 backtest
2. 使用 `smoke-positive-strategy-flow.py` 自动命中可发布样本

---

### 7.4 DashScope 调用失败

检查：

1. `DASHSCOPE_API_KEY` 是否有效
2. `LLM_PROVIDER=dashscope` 是否生效
3. `DASHSCOPE_BASE_HTTP_API_URL` 是否正确
4. 网络连通性与额度

---
