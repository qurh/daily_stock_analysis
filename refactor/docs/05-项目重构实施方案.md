# 项目重构实施方案

更新时间：2026-02-13  
文档状态：Draft v1

---

## 1. 方案目标

本方案用于指导从现有项目向重构目标架构平滑迁移，确保：

1. 原能力不丢失：分析、回测、通知、多入口交互全部保留。
2. 新能力可落地：RAG With Memory、知识学习、自我进化、流程编排、Prompt 治理。
3. 实施可控：分阶段上线、可回滚、可观测、可验收。
4. 文档先行：每次迭代有记录、有结果、有风险与决策沉淀。

---

## 2. 输入基线

实施方案基于以下已确认文档：

1. `refactor/docs/01-全量保留功能清单.md`
2. `refactor/docs/02-需求分析.md`
3. `refactor/docs/03-架构设计文档.md`
4. `refactor/docs/04-详细逻辑设计文档.md`

关键实施约束：

1. 向量库默认 `Chroma`，预留 `Milvus` 迁移能力。
2. 编排引擎默认 `LangGraph`，通过 `WorkflowEngineAdapter` 解耦。
3. 优化触发“事件驱动为主”，支持手动与 chatbot 发起；实施前必须通过评估与人工确认。
4. 前端独立重建，前后端完全解耦。
5. 回测能力为 MVP 硬约束（历史口径 F31/F32）。

---

## 3. 实施总策略

## 3.1 迁移策略

采用“并行重构 + 渐进替换（Strangler Pattern）”：

1. 在 `refactor/` 内构建新系统，不改动原目录核心逻辑。
2. 新链路先实现“可运行、可对比、可回滚”。
3. 按模块逐步替换旧链路能力，保留兼容适配层直至新链路稳定。

## 3.2 交付节奏

1. 每个阶段都包含：需求确认 -> 设计 -> 开发 -> 测试 -> 验收 -> 迭代记录。
2. 每个阶段输出明确“质量闸门”，未过闸门不得进入下一阶段。
3. 关键风险（数据源稳定性、策略劣化、流程变更）通过灰度和回滚机制控制。

## 4. 工作分解结构（WBS）

## 4.1 工程底座（WP-00）

1. `refactor/backend` 与 `refactor/frontend` 工程初始化。
2. 配置中心、日志、错误码、统一响应规范、CI 基线。
3. OpenAPI v2 草案与契约测试基础。

## 4.2 编排与 Prompt 中心（WP-01）

1. WorkflowEngineAdapter（LangGraph）落地。
2. FlowTemplate 管理、执行、回放、取消。
3. PromptTemplate 版本管理、评估、发布、回滚。

## 4.3 分析域重构（WP-02）

1. 技术面能力迁移与 TA-Lib 扩展入口。
2. 宏观、信用、情绪数据适配器。
3. 决策仪表盘统一 schema 输出。

## 4.4 知识库与记忆（WP-03）

1. Markdown 导入管道（清洗、分块、摘要、向量化）。
2. RAG 检索接口与证据回溯。
3. 会话记忆压缩与长期记忆存储。

## 4.5 认知沉淀与策略提炼（WP-04）

1. chatbot 对话核心认知抽取。
2. CognitionMemo 审核入库流程。
3. StrategyArtifact 提炼、版本、绑定编排与 Prompt。

## 4.6 回测与优化闭环（WP-05）

1. 回测服务迁移与新报告兼容。
2. 反馈采集与特征集生成。
3. 事件驱动优化任务 + 手动触发 + chatbot 提案。
4. 评估闸门、灰度发布、回滚控制。

## 4.7 通知中心插件化（WP-06）

1. 全渠道能力保留与插件化接口统一。
2. 渠道差异化渲染策略（长度、方言、分片）。
3. 发送链路可观测与失败重试。

## 4.8 交互层与前端重建（WP-07）

1. Chat、知识库、编排、策略、回测控制台页面。
2. 与后端 API 契约联调。
3. 权限与操作审计（预留单用户模式扩展）。

## 4.9 治理与可观测（WP-08）

1. Trace、Metrics、Audit 三位一体。
2. 质量看板（成功率、时延、检索质量、回测分数、反馈趋势）。
3. 发布策略治理与审批机制。

## 4.10 发布与运维（WP-09）

1. dev/staging/prod 环境定义与部署脚本。
2. 版本策略、发布说明、回滚流程。
3. 运行手册与故障处理清单。

## 5. 阶段计划与里程碑

## 5.1 Phase 0：工程初始化与基线固化

目标：

1. 建立可开发、可测试、可发布的基础工程。

主要交付：

1. `refactor/backend`、`refactor/frontend` 初始化。
2. CI 语法检查、单测任务、代码风格检查。
3. 错误码规范与基础日志规范。

质量闸门：

1. 基础流水线通过率 100%。
2. 核心脚手架有最小可运行示例。

## 5.2 Phase 1：编排 + Prompt + 分析主链路

目标：

1. 建立“可执行的分析流程图”最小闭环。

主要交付：

1. Workflow 执行引擎（LangGraph Adapter）。
2. Prompt 模板渲染与版本管理。
3. 个股分析报告输出（结构化仪表盘）。

质量闸门：

1. `flow_version + prompt_version` 在每次执行可追溯。
2. 单流程 P95 时延满足基线。

## 5.3 Phase 2：知识库 + 记忆 + Chat

目标：

1. 打通聊天问答与知识引用链路。

主要交付：

1. MD 一键导入知识库。
2. RAG 检索与证据返回。
3. 对话摘要压缩和长期记忆。

质量闸门：

1. 引用证据命中可回溯。
2. 长会话 token 控制有效。

## 5.4 Phase 3：认知沉淀 + 策略提炼 + 回测闭环

目标：

1. 实现系统自我学习与策略进化核心链路。

主要交付：

1. CognitionMemo 生成、审核、入库。
2. StrategyArtifact 版本化管理。
3. 回测评估接入策略生效闸门。
4. 反馈与优化事件驱动任务。

质量闸门：

1. 不通过评估的策略不能发布。
2. 回测结果可直接反哺优化任务。

## 5.5 Phase 4：通知插件化 + 治理完善

目标：

1. 保障生产可用性、可观测性与可运营性。

主要交付：

1. 全渠道通知插件化改造完成。
2. 质量看板、审计日志、灰度回滚流程打通。

质量闸门：

1. 渠道回归通过率达标。
2. 关键变更全量审计可追踪。

## 5.6 Phase 5：前端独立重建与上线收口

目标：

1. 形成完整的前后端解耦交付版本。

主要交付：

1. Chat、知识库、编排、策略、回测五大页面。
2. 契约联调完成并通过 E2E 回归。

质量闸门：

1. 关键业务链路全量可用。
2. 发布手册与回滚手册可执行。

---

## 6. 质量闸门（阶段共用）

每阶段必须满足：

1. 需求映射完整：FR/NFR 与实现、测试可追踪。
2. 自动化验证通过：语法、单元、集成、关键 E2E。
3. 可观测性就绪：日志、指标、trace 可查。
4. 回滚策略有效：有上一稳定版本恢复路径。
5. 文档同步完成：变更说明 + 迭代记录。

---

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 外部数据源波动 | 分析结果不稳定 | 多源容灾 + 质量评分 + 降级 |
| 编排复杂度提升 | 故障定位困难 | 节点级 trace + 统一错误码 |
| Prompt/策略劣化 | 输出质量下降 | 离线评估 + 灰度发布 + 快速回滚 |
| Chroma 性能瓶颈 | 检索时延升高 | 提前预留 Milvus 迁移脚本与适配层 |
| 前后端重构联调风险 | 交付延期 | OpenAPI 契约先行 + Mock 联调 |
