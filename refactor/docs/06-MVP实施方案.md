## 1. MVP 目标

在可控周期内交付“可用、可验证、可迭代”的首个智能股票助手版本，满足以下核心目标：

1. 用户可通过 chatbot / Web 与系统进行多轮交互。
2. 系统可进行知识导入、RAG 检索、记忆压缩、结构化分析输出。
3. 回测能力必须可用并纳入优化闭环（硬约束）。
4. 通知全渠道能力保留且采用插件化实现。
5. 支持事件驱动优化任务，并保留手动触发与 chatbot 提案入口。

## 2. MVP 范围定义

## 2.1 In Scope（MVP 纳入）

### 2.1.1 架构底座

1. Workflow Engine（LangGraph Adapter）最小可用版。
2. Prompt Center 基础版（版本管理 + 发布 + 回滚）。
3. 统一错误码、日志、trace、审计基础能力。

### 2.1.2 核心业务功能

1. Markdown 一键导入知识库（上传、清洗、分块、摘要、向量化、检索）。
2. 多轮聊天 + RAG With Memory（短期记忆 + 长期记忆基础版）。
3. 个股分析主链路（技术面 + 新闻情绪 + 宏观/信用最小因子接入）。
4. 决策仪表盘统一输出 schema。
5. 回测服务（任务执行、结果查询、汇总指标）完整可用。
6. 反馈采集与优化触发（事件驱动 + 手动 + chatbot 提案）。
7. 通知中心插件化并保留现有渠道。

### 2.1.3 认知沉淀与策略闭环（MVP 基础版）

1. 对话核心认知提炼为 `CognitionMemo` 草稿。
2. 人工审核通过后入知识库。
3. 策略提炼结果可绑定到流程/Prompt 并触发回测评估。

## 2.2 Out of Scope（MVP 不纳入）

1. 自动实盘下单与券商交易执行。
2. 全量高级策略编排可视化编辑器（先提供配置化与基础界面）。
3. 多租户/多组织权限体系。
4. Milvus 生产集群部署（MVP 使用 Chroma）。

## 2.3 Post-MVP 延期项（明确边界）

| 需求 ID | 延期原因 | 目标里程碑 |
|---|---|---|
| `FR-STR-002`（交易策略集提炼） | MVP 先打通分析策略提炼与闭环，避免策略域范围过大影响首发节奏 | `MVP+1` 迭代 |
| `FR-KB-005`（知识条目管理高级能力） | MVP 优先保障导入和检索主链路 | `MVP+1` 迭代 |
| `FR-CHAT-005`（记忆策略高级配置） | MVP 先提供默认策略，后续开放高级可配置项 | `MVP+1` 迭代 |
| `FR-FDB-004`（周期性知识总结任务） | MVP 优先事件驱动优化，周期任务次阶段补齐 | `MVP+1` 迭代 |

1. 关系库：SQLite（dev）/ PostgreSQL（staging 可选）

## 5. MVP 里程碑计划

## M0：工程与契约初始化

输出：

1. 后端与前端独立工程脚手架。
2. OpenAPI v2 初版、错误码枚举、日志规范。
3. CI 基线（语法 + 单测 + lint）。

验收：

1. 工程可运行、可测试、可部署。

## M1：编排 + Prompt + 分析最小闭环

输出：

1. FlowTemplate 执行、取消、trace。
2. Prompt 版本管理与回滚。
3. 个股分析输出仪表盘 schema。

验收：

1. 端到端完成一次分析并可追踪版本。

## M2：知识库 + Chat + Memory

输出：

1. MD 导入入库链路。
2. chat API（多轮会话 + RAG 引用）。
3. 会话摘要与长期记忆写入。

验收：

1. 问答结果含证据引用，长会话无明显性能失控。

## M3：回测 + 反馈 + 优化触发

输出：

1. 回测任务与报表 API。
2. 反馈采集与特征生成。
3. 事件驱动优化、手动触发、chatbot 提案入口。

验收：

1. 完成一次“策略提案 -> 回测评估 -> 人工确认 -> 生效/回滚”演练。

## M4：通知插件化 + 前端首版

输出：

1. 全渠道通知插件化分发。
2. 前端五大页面首版：聊天、知识库、流程、策略、回测。

验收：

1. 关键页面联调可用，通知回归通过。

## M5：稳定性收口与发布

输出：

1. 缺陷收敛、性能调优、运行手册。
2. MVP 发布包与发布说明。

验收：

1. MVP 达到发布门槛并通过回归。

## 6. MVP 测试与验收标准

## 6.1 功能验收清单

1. 文档导入后可在聊天中被检索并引用。
2. 多轮对话上下文一致，旧消息自动压缩。
3. 分析输出包含可回测字段且回测 API 正常。
4. 反馈可触发优化任务并进入评估闸门。
5. chatbot 可提交优化提案，且默认需要人工确认才实施。
6. 通知全渠道发送可回归通过。

## 6.2 指标门槛（建议）

1. 聊天接口 P95：`< 8s`（标准负载）。
2. 分析任务成功率：`>= 95%`（排除外部依赖不可用）。
3. 回测任务成功率：`>= 98%`（数据不足单独统计）。
4. 关键链路可观测覆盖率：`100%`（trace + 错误码 +审计）。

---

## 7. 发布策略与回滚

## 7.1 发布策略

1. 采用灰度发布：先内部流量，再全量。
2. Prompt/Flow/Strategy 变更先离线评估再灰度。
3. 前端按 feature flag 控制模块上线。

## 7.2 回滚策略

1. 应用版本回滚到上一稳定镜像。
2. Prompt/Flow/Strategy 回滚到上一个 `ACTIVE` 版本。
3. 数据层采用“前向兼容迁移 + 可逆脚本”。

---

## 8. MVP 运行与运维清单

上线前必须具备：

1. 配置核验脚本（模型、数据源、通知渠道、向量库）。
2. 健康检查接口与告警规则。
3. 关键问题 runbook（数据源故障、向量检索失败、消息发送失败、回测失败）。

## 9. 迭代记录要求（MVP 期间强制）

所有 MVP 迭代必须在以下目录留痕：

1. `refactor/docs/迭代开发记录/`

记录最小字段：

1. 迭代目标与范围
2. 完成项与未完成项
3. 验证结果（命令与结论）
4. 风险与阻塞
5. 决策与变更
6. 下一迭代计划

---

## 10. MVP 完成定义（MVP DoD）

同时满足以下条件视为 MVP 完成：

1. 本文第 2 节 In Scope 功能全部可用。
2. 回测能力完整可用并接入优化闭环。
3. chatbot 提案链路可用且具备人工确认闸门。
4. 通知全渠道保留且插件化实现生效。
5. 前后端独立工程可稳定发布，关键链路通过回归。
